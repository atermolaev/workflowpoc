Этот файл содержит инструкции и контекст проекта для Claude (Anthropic) при работе над Proof of Concept (PoC) системы оркестрации контент-продакшена.

## Проект: AI-оркестратор контент-продакшена (PoC)

Ключевая цель: подтвердить сценарий "Бриф → Публикация" с двумя ролями (Заказчик, Специалист) и AI-декомпозицией задач.

## 1. Основные роли и персоны

### Заказчик

- Инициирует производство, заполняет бриф, получает готовую публикацию.
- Отслеживает статусы задач и публикации.
- Может принять работу или отправить на доработку с комментарием.

### Специалист (объединяет дизайнера, писателя, верстальщика)

- Видит список задач, отсортированный по приоритету.
- Имеет два режима работы:
  - **Режим A (через AI)**: даёт указания AI-агенту для правки черновика.
  - **Режим B (ручной)**: загружает готовый материал.
- Нажимает «Готово» после завершения работы.

## 2. Ключевой E2E пайплайн

1.  **Заказчик** заполняет и отправляет бриф.
2.  **AI-агент** декомпозирует бриф на атомарные подзадачи (с зависимостями и приоритетами).
3.  **AI-агент** генерирует черновик для каждой подзадачи.
4.  **Workload Balancer** назначает исполнителя на каждую подзадачу (на основе типа задачи, роли и текущей нагрузки).
5.  **Специалист** видит задачу с черновиком и работает в режиме A или B.
6.  **Специалист** нажимает «Готово».
7.  **Assembler** склеивает все готовые подзадачи в публикацию.
8.  **Заказчик** получает публикацию на ревью → принимает или отправляет на доработку.
9.  При доработке **AI-оркестратор** анализирует комментарий, определяет затронутые подзадачи и возвращает их исполнителю.
10. Цикл повторяется до публикации.

## 3. Архитектурные решения и ключевая логика

### 3.1. Декомпозиция брифа AI

- **Входные данные брифа**: тип контента, тема, ЦА, дедлайн, вложения, ссылки на ассеты.
- **Правила декомпозиции**:
  - Подзадача атомарна для одного специалиста.
  - Имеет тип (`text`, `design`, `layout`).
  - Может иметь зависимости (блокирующие задачи).
  - Приоритет: на основе дедлайна и позиции в зависимостях.
- **Пример типов контента и подзадач**:
  - Статья: текст, иллюстрации, вёрстка.
  - Баннер: текст/слоган, визуал.
  - Видео: сценарий, сториборд, монтаж.
  - Email: текст, вёрстка шаблона.

### 3.2. Логика редактирования через AI (режим A)

- **Промпт-цепочка**: AI работает с контекстом:
  `[Исходный бриф] + [Тип подзадачи] + [Текущая версия черновика vN] + [Инструкция специалиста] → [vN+1]`
- **Версионирование**: каждая правка создаёт новую версию (`v1 → v1.1`).
- **Diff и откат**: UI показывает изменения, возможен откат к любой версии.
- **Лимит итераций в PoC**: 5 на подзадачу, затем предложение ручного режима.
- **Проверка зависимостей**: при критическом изменении (длина текста, пропорции) → флаг "⚠️ Может повлиять на зависимые задачи".

### 3.3. Выбор исполнителя (Workload Balancer)

- **Данные об исполнителе**: `user_id`, `roles`, `status` (available/vacation/sick), `current_load`.
- **Алгоритм**:
  1. Определить тип задачи.
  2. Найти пользователей с подходящей ролью и статусом `available`.
  3. Рассчитать `load` = количество задач в статусах `InProgress` или `In_Review`.
  4. Назначить кандидата с минимальной нагрузкой.

### 3.4. Работа с медиаактивами

- **Метаданные ассета**: `asset_id`, `type`, `category`, `brand`, `tags`, `dimensions`, `license`, `created_by`, `linked_briefs`.
- **Сценарии**:
  - **Чтение**: поиск по `tags`, `category`, `brand` при создании брифа.
  - **Запись**: новые ассеты сохраняются с метаданными из контекста (`brief_id`, `category`, `brand`).
  - **Переиспользование**: AI может предложить существующие ассеты.
- **API хранилища**:
  - `GET /assets?tags=...&category=...` — поиск.
  - `GET /assets/{id}/download` — скачивание.
  - `POST /assets` — загрузка с метаданными.
  - `PATCH /assets/{id}` — обновление метаданных.

## 4. Статусная модель и загрузка

### 4.1. Статусы задач и их влияние на `current_load`

| Статус             | Входит в загрузку       | Пояснение                     |
| ------------------ | ----------------------- | ----------------------------- |
| Created            | ❌ Нет                  | Ещё не назначена              |
| Assigned           | ❌ Нет                  | Назначена, но не взята        |
| **InProgress**     | ✅ **Да**               | Исполнитель работает          |
| AI_Editing         | ❌ Нет (или ×0.5)       | AI работает, исполнитель ждёт |
| Manual_Upload      | ✅ Да                   | Исполнитель загружает файл    |
| Done               | ❌ Нет                  | Завершена                     |
| **In_Review**      | ✅ **Да**               | На проверке у специалиста     |
| Revision_Requested | ✅ Да (если возвращена) | Доработка                     |
| Assembled          | ❌ Нет                  | Склеена, ждёт заказчика       |
| Published          | ❌ Нет                  | Опубликована                  |

### 4.2. Диаграмма переходов (основные)

`[*] → Created → Assigned → InProgress ⇄ AI_Editing/Manual_Upload → Done → Assembled → In_Review ⇄ Revision_Requested → Published`

## 5. Требования к реализации (приоритеты)

### P0 — Must Have (для PoC)

- Форма брифа с подключением ассетов.
- AI-декомпозиция на подзадачи с зависимостями.
- Автоназначение исполнителя (роль + нагрузка).
- Список задач специалиста по приоритету.
- Два режима: AI-правки (с diff/откатом) или ручная загрузка.
- Кнопка «Готово».
- Интерфейс ревью с комментированием (конкретный блок или вся публикация).
- Роутер правок: комментарий → исполнитель (явная привязка + AI-анализ).
- Склеивание частей в публикацию.
- Дашборд заказчика (количество задач, статусы, версия публикации).
- Сохранение новых ассетов в хранилище с метаданными.

### P1 — Nice to Have

- AI-предложение существующих ассетов при декомпозиции.
- Нотификации при смене статуса.
- Интеграция с Sigma.
- История правок по подзадаче.
- Лимит циклов доработки с эскалацией.

### P2 — Future

- Роль Руководителя (аналитика, бюджет).
- Авторизация через внутреннюю систему, RBAC.
- S3-адаптер для хранилища.
- BEM Platform рендеринг.

## 6. Не-цели PoC (что НЕ нужно реализовывать)

- Авторизация через внутреннюю систему.
- Полноценный UI с дизайн-системой.
- Роль Руководителя.
- Автопубликация на сайт.
- Миграция исторических задач.

## 7. Технические требования к AI-агенту (для режима A)

- Поддержка `system prompt` с фиксированным контекстом брифа.
- Режим `instruct/edit`, а не генерация с нуля.
- Возврат результата в том же формате, что и черновик (Markdown, JSON с URL, SVG/PNG).
- Логирование: `brief_id → task_id → version → instruction → result_hash`.
- Интеграция с медиахранилищем и пайплайнами генерации.

## 8. API ожидания (для бэкенда/интеграций)

### Хранилище ассетов

- `GET /assets?tags=&category=&brand=`
- `GET /assets/{id}/download`
- `POST /assets` (multipart/form-data + metadata JSON)
- `PATCH /assets/{id}`

### Пользователи и нагрузка

- Структура пользователя: `{ user_id, name, roles, status, current_load }`
- `current_load` пересчитывается динамически на основе статусов задач.

- **Для заказчика**: форма брифа, дашборд со статусами задач, интерфейс ревью публикации.
- **Для специалиста**: список задач (сортировка по приоритету), детальный вид задачи (бриф + черновик + два режима работы), предпросмотр, кнопка «Готово».
- **Общие элементы**: отображение diff при AI-правках, возможность отката версий, комментарии к доработке.

## 11. Файловая структура проекта.

- Для каждого ReactJS-компонетна должна быть отдельная папка.
- Все компоненты должны лежать в папке /src/components/\*
- все утилитарыне функции, типы, и общие вещи, которые используются в большинстве компонентов, должны быть в папке globals, она расположенна здесь: /src/globals/
- Типы конкретного компонента должны быть вынесены в отдельный файл расположенный в папке с компонентом.

## 12. Архитектура проекта

- Для хранения состояний использовать redux.
- Для реализации взаимодействия ReactJS и redux использовать пакет reduxjs/toolkit.
- Все файлы с редаксом лежат в папке /src/redux/
- Для хранения состояний использовать createSlice. Они должны лежать в папке /src/redux/slices/
- Для обращения к backend по rest api использовать createApi, из пакета reduxjs/toolkit. Этот код должен лежать в папке /src/redux/api/
- Билд проекта должен лежать в папке /public/\*
